Why it works
Architecture

Structured knowledge, not scattered threads.

⚡
Your morning briefing, automated
One summary file captures everything your AI needs before it starts: what is active, what is blocked, who owns what, what was just decided. Your first message of the day goes straight to the work, not the catch-up.
Day 1
your AI already knows your full team context
📋
Nothing falls through the cracks
Every decision is captured: what was decided, by whom, and when. Open items stay visible until closed. Articles that go stale are flagged automatically, so your AI never confidently gives you yesterday's answer.
Zero
decisions lost to a Slack thread scrolling past
🔔
Built from what your team already writes
Slack, meeting notes, Notion, Google Docs, email threads. The wiki is generated from your existing communications. Nothing to fill in manually, no new habit to form. Just point it at the right place.
9+
tools it can read your team's history from

Built for token efficiency and agent readability.

⚡
state.md as a compressed snapshot layer
Without state.md, agents fetch index.md + all workstream and people files at Step 0. On a mature wiki with 10+ workstreams, that is 30-40K tokens before any live data is pulled. With state.md, Step 0 costs 800-1,500 tokens. Individual files are fetched only when a specific gap forces it.
98%
token reduction at Step 0 vs naive full-index loading
📋
YAML frontmatter + backlink schema
Every article carries summary, status, and last_updated in YAML frontmatter. Agents orient by scanning frontmatter only. The staleness rule surfaces aging context without agents needing to diff timestamps manually.
7 days
staleness threshold, then [STALE] is injected into index.md
🔔
Source-agnostic via MCP
The bootstrap prompt is parameterized around sources, not format. Slack MCP is the default but any chronological data source works: Apple Notes MCP, Google Drive MCP, Notion MCP, pasted transcripts. The key requirement is a chronological record of decisions.

Structure

Everything your team knows, organized.

Git-native, markdown-first, agent-optimized.

Projects, key people, active experiments, decisions made, and metrics that matter. Each article is short, current, and linked to related ones. Your AI navigates it like a codebase.

Each article is a plain markdown file with YAML frontmatter. Backlinks are wikilink-style [[path/file]] references. The full repo is a git history, so agents can diff, track, and audit any change. Multiple contributors work via normal pull requests.

The summary file is the key
state.md regeneration contract
One auto-updated summary captures everything: current project statuses, open tasks, who owns what, active experiments, and key numbers. Your AI reads this first and usually that is all it needs.
state.md is always a full rewrite, never appended. Agents regenerate it completely at the end of every run that produces new decisions or status changes. It is never manually edited. This guarantees freshness and makes it safe to cache aggressively at Step 0.

Quickstart

Up and running in four steps.

1
Copy the template
Click "Use this template" on GitHub to create your own copy of the repo. GitHub walks you through it in under a minute.
2
Tell it about your team
Open one config file and fill in your name, your team's projects, the people you work with, and where your updates live: Slack, meeting notes, a Google Doc, whatever you use.
3
Run the setup with Claude
Not sure how? Just click the button below. Claude will guide you through every step. By the end, your AI will have full context of your team and be ready to start tomorrow morning.
Set it up with Claude
4
Your AI has context from now on
Add one line to any AI workflow telling it to read your wiki first. Every task starts with a full picture of your team: who owns what, what has been decided, what is in progress.
1
Clone and reinitialize
Fork via the template button, or clone directly and start a clean git history.
$ git clone https://github.com/bkocdur/agent-wiki-template my-wiki
$ cd my-wiki && rm -rf .git && git init
2
Configure CLAUDE.md
Replace the owner block with your name, role, and company. Set your Slack channel ID (or swap for another source). Adjust workstream and people lists to match your team structure.
3
Fill in and run BOOTSTRAP.md
Set your data source, date range, workstreams, and people in BOOTSTRAP.md. Open Claude Code in the wiki directory and paste the file contents. The agent fetches, extracts, writes, and commits.
$ cd my-wiki
$ claude
# paste contents of BOOTSTRAP.md and send
4
Add Step 0 to your agent prompts
Paste this block at the top of any agent workflow. state.md is your compressed context layer. Agents regenerate it at the end of every run that produces new decisions.
Step 0 - Wiki context: Fetch state.md from your wiki repo.
Use it as baseline before pulling live data.
Regenerate state.md after every run producing new decisions.
